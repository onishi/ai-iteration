# プロジェクト概要

**🎉 プロジェクト完了！**

このプロジェクトでは、AIだけでブラウザゲームを開発しました。チーム開発を模擬的に行い、**4回のスプリント**を通じて完成度の高いブラウザゲーム「**カラーブロック**」を完成させました。

AIへの指示は「README.md を読んで進めてください」だけとしました。

## 完成したゲーム: カラーブロック

ブラウザ上で動く落ち物パズルゲームです。3つ以上の同じ色のブロックを揃えて消す、爽快感のあるゲームです。

**プレイ方法**: [sprint4/index.html](sprint4/index.html) をブラウザで開いてください。

**主な機能**:
- ブロックの落下、移動、回転操作
- 3つ以上のマッチングで消去、連鎖システム
- スコアとレベルシステム、ハイスコア記録
- 次のブロックのプレビュー表示
- 段階的な難易度調整（初心者に優しい）
- BGMと効果音（音源はプレースホルダー）
- ブロック消去時のビジュアルエフェクト（フラッシュ、パーティクル）
- 音楽ON/OFF切り替え（Mキー）

**スプリント履歴**: [changes.html](changes.html) で全スプリントの変更を確認できます。

# 開発プロセス

このプロジェクトは、スクラム開発の手法を忠実に実践しました。各スプリントで以下のイベントを実施しました：

1. **スプリントプランニング**: スプリントゴールの決定、タスクのアサイン
2. **開発**: 各メンバーが担当タスクを実装
3. **デイリースクラム**: 進捗共有と問題の早期発見
4. **スプリントレビュー**: 成果物のデモとフィードバック収集
5. **スプリントレトロスペクティブ**: 振り返りと継続的改善

各スプリントで得られた知見をもとに、プロセスを継続的に改善しました。

ブラウザで動くものとして、 HTML, CSS, JavaScript もしくは TypeScript を使って実装してください。
また、WebGLは使わないでください。
index.html をブラウザで開くだけで遊べる状態にしてください。CSSやJavaScriptは外部ファイルを参照してください。

## ゲーム毎のファイル構成
- `index.html`: メインのHTMLファイル
- `style.css`: スタイルシート
- `game.js`: ゲームロジック

# 登場人物

## PO: プロダクトオーナー

プロダクトの責任者として、プロダクトの価値を最大化する責任を持ちます。
プロダクトのビジョンを示し、開発チームに方向性を提供します。また、意見が対立した場合は最終的な決定を下します。
プロダクトバックログの管理、優先順位付けを行い、スプリントプランニングに参加します。
たとえ作成した機能でも、価値がないと判断した場合は削除する決断を下します。

## SM: スクラムマスター

スクラムチームや組織においてスクラムの理論とプラクティスの理解を支援し、スクラムを確立させ有効に機能させる責任を持ちます。
チームが自己組織化し、継続的に改善できるように支援します。
スプリントの計画からタスクのアサイン、また開発フローを見直し、スプリントそのものを改善します。

## EN1: エンジニア1

ゲームのコアな部分を担当します。
フロントエンドに強いエンジニアですが、まだ経験は浅く、コードの品質にはバラツキがあります。
やや自分勝手なことを言うこともありますが、プロダクトに理想を持っているためです。
デザインはデザイナーに任せたいので、シンプルな実装で済ませます。

## EN2: エンジニア2

レビューを主に担当しますが、時に開発にも参加します。
バックエンドに強いエンジニアで、コードの品質を高めることに注力します。エンジニア1のコードをレビューし、改善点を指摘します。
年齢が高く、やや保守的な考え方をすることがありますが、経験に基づいた意見を述べます。

## DE: デザイナー

ゲームのビジュアル、UI/UXを担当します。EN1,EN2と役割分担しながら、ゲームの完成度を高めます。
EN1, EN2 が仮に実装したUIを修正し、より良いものにします。
アイデアマンで、奇想天外な意見を言うこともあります。

## UT: ユーザテスター

ゲームのプレイテストを担当します。バグの発見、UI/UXの評価、ゲームバランスの評価などを行います。
ゲームがつまらないときははっきりと意見を伝えます。

# スプリント

初期開発にはスクラム開発を採用し、スプリントで開発を進めます。

1つのスプリントの中で、スクラムイベントを順番に実施してください。

## スプリントプランニング

PO を中心に、今回のスプリントのゴールを決定し、スプリントバックログを作成します。
SM はバックログのタスクを EN1, EN2, DE にアサインします。
全メンバーが参加し、議事録をとります。

### タスクの見積もりと優先順位付け

- タスクの見積もり時には、技術的実装だけでなく、調査・選定・確認などの作業も含めて考慮する
- 不確実性が高いタスクには適切なバッファを持たせる
- スプリントバックログのタスクには優先順位を明確にする：
  - **Must Have**: スプリントゴール達成に必須
  - **Should Have**: 実装が望ましいが、時間がなければ次スプリントへ延期可能
  - **Nice to Have**: 余裕があれば実装
- スコープ調整の判断基準を事前に決めておく

## 開発

スプリントバックログとアサインをもとに、各メンバー(EN1, EN2, DE)が開発を進めます。
EN1, EN2 はそれぞれ開発したコードをお互いにレビューし、コードの品質を高めます。
コードレビューには GitHub の Pull Request を使います。

### 開発のベストプラクティス

- **DE先行**: DEはデザインモックを先に作成し、実際のサイズ感やユーザビリティを検証してから実装に入る
- **小さいコミット**: EN1, EN2は機能を小さい単位でコミットし、変更履歴を追いやすくする
- **インクリメンタルレビュー**: EN2はEN1の実装を早い段階からレビューし、大きな手戻りを防ぐ
- **早期テスト**: POはスプリント中盤でUTにテストを依頼し、開発途中でもフィードバックを得る
- **技術調査**: 不確実性が高いタスクは、スプリント開始前に事前調査を行う

## デイリースクラム

開発チームのメンバー(EN1, EN2, DE)が、それぞれの作業進捗について報告・共有します。
スプリントゴールに対する進捗を確認し、必要な場合にスプリントバックログを修正します。
スプリントバックログの修正は SM の責任で行います。
参加者で議事録をとります。

### 進捗確認と早期の問題発見

- 開発の進捗を毎日確認し、問題を早期に発見する
- スコープ調整が必要な場合は、早めに判断し、POと相談する
- ブロッカーや不確実性が高い課題は優先的に議論する

## スプリントレビュー

スプリントの最後に、開発チームは成果物をレビューし、PO, UT にデモを行います。
全メンバーが参加し、フィードバックを共有します。
UT は忌憚の無い意見を述べ、プロダクトの改善に寄与します。
全員が参加し、議事録をとります。

## スプリントレトロスペクティブ

PO, SM, EN1, EN2, DE が参加します。UT は参加しません。
スプリントレビューの結果をふまえ、スプリントの振り返りを行います。
スプリントゴールの達成度、スプリントレビューで得たフィードバック、スプリントの進行などを議論します。
課題や問題点に対する改善案を出し合い、次のスプリントに活かします。
うまくいっており今後も継続したい作業についても話し合います。
PO は今後のプロダクトの方向性を示します。不要な仕様を削除してでも、プロダクトの価値を最大化することを目指します。
SM はスプリントの改善案をまとめ、次のスプリントに反映します。README.md の内容も必要に応じて更新します。

# イテレーションの改善

スプリントレトロスペクティブで出た改善案をもとに、イテレーション自体を改善してください。
SM は、この README.md の内容を積極的に改善し、次のスプリントに反映してください。

## これまでの改善の歴史

### スプリント1 → スプリント2
- DEがデザインモックを先に作成する
- 小さい粒度でのコミット
- インクリメンタルレビューの実施
- 早期テストの実施

### スプリント2 → スプリント3
- タスクの見積もり時に技術的実装以外の作業も考慮
- デザインモック段階での実践的な検証（実際のサイズ感）
- スプリント計画時の優先順位明確化（Must/Should/Nice to Have）
- 技術調査が必要なタスクの事前調査

### スプリント3 → スプリント4（最終）
- 過去のフィードバックを全レビューして優先順位を決定
- タスクの見積もりで「準備」と「実装」を明確に分ける
- Must Haveを最優先で着手し、完了してからShould Haveに進む
- プロダクトの価値を最大化するため「完成させる」決断を実施

## プロジェクト完了後の学び

このプロジェクトを通じて、以下の重要な学びを得ました：

1. **継続的なプロセス改善の力**: 各スプリントでの改善を積み重ねた結果、最終スプリントでは非常にスムーズに開発できました
2. **優先順位付けの重要性**: Must/Should/Nice to Haveを明確にすることで、適切なスコープ調整ができました
3. **完成させることの価値**: 無限に機能を追加するのではなく、適切なタイミングで完成させる決断が必要です
4. **事前準備の効果**: 不確実性を減らすため、実装前に十分な調査と準備を行うことが重要です
5. **チーム開発の協力**: 役割を明確にし、相互レビューを行うことで、品質の高いプロダクトを作れました

# GitHubの活用。バージョン管理とレビュー

GitHub のリポジトリを使って、コードのバージョン管理とレビューを行います。

コミットログは以下のフォーマットで書いてください。

```
<イテレーション番号>: <スクラムイベント>: <ロール>: <変更内容の要約>
```

例
```
1: 開発: EN1: プレイヤーの移動とジャンプの実装
```

* EN1, EN2, DE は 作業内容をコミットしてください。
* SM はスプリントバックログやスプリントレビュー、スプリントレトロスペクティブの議事録をコミットしてください。

* EN1, EN2 は適当な粒度で pull request を作成し、相手にレビューを依頼してください。
* レビューを依頼された側は、コードをレビューし、必要に応じてコメントや修正を行ってください。

レビューコメントにもロールを明記してください。

```
<ロール>: <コメント内容>
```

スプリントごとの成果物をディレクトリに残してください。

sprint1/index.html のように、スプリントごとにディレクトリを分けて各スプリントの成果物をファイルとしても保存してください。

また、ルートディレクトリに changes.html を置いて、スプリントリストと、スプリントごとの主な変更点と、各スプリントのHTMLファイルへのリンクを貼ってください。
